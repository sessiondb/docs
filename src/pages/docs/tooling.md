# SessionDB Tooling (`scli`)

The **SessionDB CLI (`scli`)** is the powerful all-in-one utility to install, configure, start, and seamlessly upgrade your SessionDB instance.

## Command Reference

| Command | Description |
|---|---|
| **`scli init`** | Starts an interactive setup for DB/Redis connections. Automatically generates crucial secrets, and saves variables to **`config.toml`** (single source of truth). Also generates `.env` for backend processes. |
| **`scli install [version]`** | Downloads the specified SessionDB artifacts. Extracts contents into root `versions/<tag>/` and updates the `current` symlink. |
| **`scli get version [workdir]`** | Analagous to install, but explicitly uses the given `workdir` as the install root. |
| **`scli run [version] [workdir]`** | Starts the SessionDB server in the background and tails logs dynamically. Picks up configuration directly from **`config.toml`**. You can safely press `Ctrl+C` to stop the log tail—the server will continue running. |
| **`scli stop`** | Sends a stop signal to gracefully kill the locally executing server background process (if originally started by `scli run`). |
| **`scli migrate`** | Automatically sends a `POST /v1/migrate` payload, passing the `X-Migrate-Token` pulled directly from the generated secrets file. |
| **`scli status`** | Validates the system heartbeat by attempting to query the underlying network health endpoint (`GET /health`). |
| **`scli deploy`** | Dynamically synthesizes Systemd units tailored for bare metal integrations. Ingests properties directly from **`config.toml`**. |
| **`scli reset`** | Reverts and removes the main SessionDB install payload directories. Use `--all` to flush configurations entirely. |
| **`scli prune --yes`** | **Destructive operation**. Flushes all trace of SessionDB artifacts permanently from the host layer, wiping configs, logs, pids, and binaries entirely. |
| **`scli update`** | Queries Github for the 5 latest `scli` revisions for a self-updating mechanism in-place. |

## Configuration Handling

SessionDB strictly relies on a single configuration standard to ensure portability and immutability across diverse bare metal layers.

1. **`config.toml` (The Source of Truth):**
   When you run `scli init`, this file is generated containing clear TOML blocks for connections: `[server]`, `[database]`, `[redis]`, `[jwt]`, `[secrets]`, `[auth]`, and `[ui]`.
   
   To edit the values safely without risking syntax issues, use:
   ```bash
   scli config edit
   ```
2. **Generated Environment (`.env`):**
   If deploying via Systemd or raw binaries, `scli` actively translates the TOML structure into an `.env` format specifically to inject shell environments effectively for the backend engine executable to interpret via `EnvironmentFile=/path/to/.env`.
3. **Storage Directories:**
   By default, profiles are pushed to `$HOME/.config/sessiondb` or paths driven by `SESSIONDB_CONFIG_DIR`. You may supply `--config-dir` to core tooling execution logic (such as `scli run` or `scli deploy`) to pivot context mapping dynamically.

## Advanced CLI Maintenance

### Systemd Seamless Upgrades
The internal logic relies heavily on the `current` symlink block mechanism. Since Systemd files (generated by `scli deploy`) target the `current` symlink directory instead of a hardcoded path, upgrading is instantaneous with zero-downtime reconfiguration required.
When you run `scli install <new-version>`, the logic transparently repoints the directory structures and triggers the system daemon layers to safely reload and execute the new processes seamlessly.

### Self-Updating `scli`
To keep the CLI up to date with core security mechanisms independently of the database schema:
```bash
scli update
```
